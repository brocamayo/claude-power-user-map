Section 02
Plan Mode & Outline-First Prompting

Never let Claude write code on an ambiguous brief. Think of yourself as the senior engineer -- Claude is the junior. Your job is to define, review, approve.

⚡
CLAUDE CODE
Use --plan flag or type /plan in session. Claude proposes changes as a structured diff preview -- no files are touched until you approve.
💬
CHAT / API
No native plan mode. Use Outline-First Prompting: ask Claude to list its approach and assumptions before writing anything. You review the outline, then say "proceed."
TRIGGER
Enter Planning Phase
  • Claude Code: claude --plan or /plan
  • Chat/API: "Outline your approach before writing any code"
  • State the problem in 1-2 sentences
  • List files/systems that will be touched
→
SCOPE LOCK
Before Any Run
  • Define explicit success criteria
  • Name files that must NOT be touched
  • Set output format expectations
  • Agree on error handling approach
→
REVIEW
After Plan Is Proposed
  • Read every step mentally before approving
  • Flag any step that's too broad
  • Ask "What could go wrong at step X?"
  • Inspect proposed edits before accepting
→
EXECUTE
After Approval
  • Watch context usage -- bail if bloating
  • Interrupt immediately if it deviates from plan
  • Commit checkpoints to git frequently
  • Test after each logical unit, not at the end
💡
Rule of thumb: If the task touches more than 2 files or takes more than 10 minutes to write manually -- planning is not optional. Ambiguous scope = expensive rewrites.
Section 03
Development Pipeline

A repeatable 6-step loop for any system build -- CLI tools, data pipelines, web apps, automation agents, analysis scripts.

📂
Step 01
Context Loading
  • Feed architecture docs & README first
  • Include relevant schemas or data models
  • Paste full error logs -- never summarize
  • Add style guides or linting rules
🧠
Step 02
Problem Framing
  • State what you have vs. what you want
  • Give a concrete example of desired output
  • Mention constraints (lang, deps, perf)
  • Say what you've already tried
📋
Step 03
Plan Approval
  • Request outline before code generation
  • Validate approach against your mental model
  • Check for unintended side effects
  • Ask: "What assumptions are you making?"
🔧
Step 04
Incremental Build
  • Build one module at a time
  • Test each module before moving on
  • Use TODOs to defer scope for next turn
  • Avoid monster prompts that do everything
🔍
Step 05
Review & Harden
  • Ask Claude to critique its own code
  • Request edge case enumeration
  • Ask: "What would break this in prod?"
  • Run a separate security review prompt
📦
Step 06
Document & Ship
  • Have Claude write inline docstrings
  • Generate handoff README or SOP
  • Log what was built and why
  • Archive the full conversation
Section 04
Subagents, Dynamic Workflows & Loops

The most powerful Claude patterns go beyond a single prompt-response. Orchestration -- chaining agents, spawning subagents, and building feedback loops -- unlocks order-of-magnitude productivity gains.

🎯
TIER 1 -- SINGLE AGENT
One Claude, one task, one session
Default
When to use:
  • Task is well-scoped and fits in one context window
  • Output is a single file, document, or answer
  • No parallelization needed
  • Quick iteration more valuable than automation
Key patterns:
  • Plan, Execute, Review in one session
  • Multi-turn refinement within one conversation
  • Ask Claude to self-critique and revise
  • Use structured outputs for downstream parsing
🔗
TIER 2 -- PROMPT CHAINING
Sequential tasks, output feeds next input
Intermediate
When to use:
  • Task has clear sequential stages
  • Each stage needs validation before next runs
  • Output of step N is input to step N+1
  • You want human checkpoints between stages
Examples:
  • Research → Outline → Draft → Edit → Publish
  • Scrape data → Clean → Analyze → Visualize
  • Define schema → Generate code → Write tests
  • API design → Implementation → Docs → Deploy

TIER 3 -- ORCHESTRATOR PATTERN
You architect the pipeline; Claude runs each node
Advanced
⚠
Important: Claude does not autonomously spawn subagents in chat or standard API use. You are the orchestrator -- you design the pipeline, run each Claude call, and pass outputs between them. In Claude Code, the Task tool can dispatch subtasks, but this is an advanced opt-in feature.
When to use:
  • Task has parallel independent subtasks
  • Different roles need different system prompts
  • Work can be divided, run separately, then recombined
  • You want specialized context per role
How to build it:
  • Call Claude with a Researcher system prompt → get output
  • Call Claude with a Builder system prompt → pass research as input
  • Call Claude with a Validator prompt → check the build
  • Aggregate results and call a final Writer prompt

TIER 4 -- DYNAMIC LOOPS
Self-correcting agents that loop until done
Power
When to use:
  • Output quality is measurable (tests pass/fail)
  • Task requires iteration to converge
  • You can define a clear exit condition
  • Errors can be fed back as new context
Loop safety rules:
  • Always set a max iteration cap (3-5 loops)
  • Log each iteration's state for debugging
  • Define what "good enough" means before starting
  • Human-in-the-loop checkpoint every N cycles

Section 05
Context Hygiene

Context management is invisible to most users and critical to all power users. The quality of Claude's output is bounded by the quality of your context.

✅ ALWAYS INCLUDE
  • Full error messages and stack traces
  • Exact file paths (not "the main file")
  • Language + runtime + library versions
  • What the code is supposed to do
  • What it's actually doing instead
⛔ NEVER DO
  • Summarize errors in your own words
  • Paste 1000+ lines without flagging key area
  • Start new session mid-problem without recap
  • Let context window bloat past ~70-80% on precision tasks
  • Assume Claude remembers previous sessions
🔄 RESET SIGNALS
  • Responses start drifting or hallucinating
  • Claude contradicts its earlier plan
  • Output quality degrades noticeably
  • You're 30+ turns deep on one problem
  • It suggests something you already ruled out
💡 PRO TECHNIQUES
  • Open each session with a context brief
  • Use Projects for persistent system prompts
  • Pin key constraints at top AND bottom of prompt
  • Use XML tags to separate context sections
  • Summarize and paste key decisions as you go
Context Window Health
Rule of thumb -- larger models handle long context better than smaller ones.
0-60% -- Optimal
60-80% -- Monitor for drift
80-100% -- Consider reset
Creative tasks tolerate long context better than precise code generation. Watch for behavioral signals (drift, contradiction) more than raw token counts.
Section 06
Best Practices

Principles that separate expert-level prompting from average usage. These are model-agnostic -- they work with any frontier LLM.

FOUNDATIONAL LEVERAGE ADVANCED UNIVERSAL
🎯
One Job Per Prompt FOUNDATIONAL
Single responsibility principle applies to prompts. Don't ask Claude to design + build + test + document in one shot.
🗃
Use Projects for Domain Work LEVERAGE
Working in the same codebase weekly? A Project with system prompt beats starting fresh every time.
🔁
Iterative Refinement Loop LEVERAGE
Generate, critique, refine is 3x better than accepting the first output. Always ask Claude to improve its own work.
📏
Explicit Output Contracts FOUNDATIONAL
"Write a function" is weak. "Write a Python function that returns X given Y, with docstring and type hints" is strong.
🧪
Test-First Prompting ADVANCED
Ask Claude to write tests before implementation. Forces understanding of acceptance criteria before writing code.
🔐
Adversarial Review Pass ADVANCED
After any generated system: "Act as a senior engineer trying to break this. What are the failure modes and security gaps?"
📌
Anchor Key Constraints FOUNDATIONAL
Repeat the most important constraints at both the beginning and end of long prompts. Context window attention is not uniform.
🌐
Model-Agnostic Patterns UNIVERSAL
Every pattern in this guide applies to GPT-4, Gemini, Mistral, and others. Strong prompt architecture transcends any one model.

Section 08
Model Selection Guide

Different models have different strengths. Picking the right model for each task saves money, reduces latency, and often produces better results than defaulting to the most powerful option.

★
Opus
Maximum Intelligence
The most capable model. Use when quality matters more than speed or cost.
  • Complex architectural decisions
  • Nuanced writing (essays, strategy docs)
  • Multi-step reasoning & math proofs
  • Ambiguous problems with many edge cases
  • Code review of critical systems
  • Research synthesis across many sources
Slowest Highest cost Best quality
◆
Sonnet
Best All-Rounder
The sweet spot for most tasks. Strong reasoning at reasonable speed and cost.
  • Day-to-day coding & debugging
  • Content drafting & editing
  • Data analysis & transformation
  • API integrations & tooling
  • Agentic workflows (Claude Code default)
  • Most prompt chaining pipelines
Balanced Default choice Great value
△
Haiku
Speed & Efficiency
Fastest and cheapest. Use for high-volume, well-defined tasks where latency matters.
  • Classification & categorization
  • Data extraction from structured text
  • Simple Q&A and lookups
  • Content moderation & filtering
  • High-volume API calls (batch processing)
  • Routing and triage in pipelines
Fastest Lowest cost High volume
Task TypeRecommended ModelWhy
Complex architecture design Opus Needs deep reasoning across many constraints and trade-offs
Security audit / code review Opus Subtle vulnerabilities require maximum attention to detail
Day-to-day feature development Sonnet Best balance of code quality, speed, and cost for routine work
Writing & content creation Sonnet or Opus Sonnet for drafts and iteration; Opus for final polish on high-stakes pieces
Data pipeline / ETL scripts Sonnet Structured transformation with clear inputs/outputs -- Sonnet handles well
Agentic loops (eval-fix, critic-refine) Sonnet Multiple iterations multiply cost; Sonnet keeps loops affordable
Classifying or tagging items Haiku Well-defined categories with clear rules -- speed wins
Extracting fields from documents Haiku Structured extraction from known formats is Haiku's sweet spot
Routing / triage in a pipeline Haiku Fast first-pass classification before sending to a bigger model
Debugging a tricky, elusive bug Opus Subtle bugs require the deepest reasoning and widest hypothesis space
Model Decision Tree
Is this a high-stakes decision with ambiguity? → Opus
Does the task require deep multi-step reasoning? → Opus
Is it a well-scoped coding or writing task? → Sonnet
Will this run in an agentic loop (multiple iterations)? → Sonnet (cost control)
Is it high-volume with a clear, simple pattern? → Haiku
Does latency matter more than nuance? → Haiku
Not sure? Start here and upgrade if quality falls short. → Sonnet (safe default)
💡
Cascade pattern: Start with Haiku for triage/routing, escalate to Sonnet for execution, use Opus only for final review or edge cases. This can cut costs 5-10x vs. running everything through Opus.
⚠
Don't default to Opus: Running every task through the most powerful model is wasteful. Sonnet handles 80% of tasks at comparable quality. Reserve Opus for the 20% that genuinely need maximum reasoning depth.
🌐
Model-agnostic equivalent: This tiering pattern applies across providers. GPT-4o maps roughly to Sonnet, GPT-4o-mini to Haiku, and o1/o3 to Opus-class reasoning. Gemini Pro/Flash follow the same split. Always match model tier to task complexity.
