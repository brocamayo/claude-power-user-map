Building with Claude

Best practices for using AI when developing apps, writing code, and shipping software.

Before You Start
📋 Set Up a CLAUDE.md
Create a CLAUDE.md file in your project root. It tells Claude your stack, conventions, and constraints every session — no re-explaining.
# CLAUDE.md # Stack: Next.js 14, TypeScript, Tailwind # DB: Postgres via Prisma # Tests: Vitest + Playwright # Rules: No any types. Always use # server actions for mutations.
📦 Feed Architecture First
Before asking Claude to build anything, share your folder structure, key schemas, and existing patterns. Context quality = output quality.
  • README + architecture docs
  • Database schema or data models
  • Existing component/module patterns
  • Linting and style rules
The Dev Workflow
1. Share context → 2. Describe the goal → 3. Ask for plan → 4. Build one piece → 5. Test it → 6. Next piece
⛔ Never say "build me a full app." Break it into features. One prompt per module. Test between each. Monster prompts produce monster bugs.
Prompting for Code
❌ Weak Prompt
Build me a user authentication system
✅ Strong Prompt
Add email/password auth to our Next.js app using NextAuth.js v5. Use our existing Prisma User model. Include: - Sign up, sign in, sign out - Protected API routes via middleware - Session provider in root layout - No OAuth for now, just credentials Write tests with Vitest.
🎯 What to include
  • Stack and framework versions
  • Specific libraries to use (or avoid)
  • Concrete inputs and outputs
  • Error handling expectations
  • What NOT to touch
🚫 What to skip
  • Vague goals ("make it better")
  • Multiple unrelated features at once
  • Assumed context from prior sessions
  • Asking for perfection in one shot
Claude Code Tips
⚙ Plan Mode
Run claude --plan or type /plan in a session. Claude proposes changes as a diff preview — nothing executes until you approve. Use for anything touching 2+ files.
🛠 Git Checkpoints
Commit after every working milestone. If Claude breaks something 5 steps later, you can roll back. Treat each commit as a save point.
🔍 Review Diffs
Always read Claude's diffs before accepting. Skim for unintended changes — removed imports, changed function signatures, deleted comments.
📊 Watch Token Usage
If Claude starts giving vague or contradictory answers, you've likely hit context limits. Start a fresh session with a recap of what's been built.
Testing & Review
🧪 Tests First
Ask Claude to write tests before implementation. Forces clear thinking about what the code should actually do.
🔄 Eval-Fix Loop
Generate → run tests → paste failures → fix. Cap at 3-5 rounds. Most things converge by round 2.
🔐 Security Review
"Act as a senior security engineer. Find every vulnerability in this code." Always do this for auth, payments, and user data.

Mistakes That Waste Time
💥 "Build the whole thing"
Asking for a complete app in one prompt produces fragile code full of shortcuts. Break it into features and build each one separately.
💥 Accepting without reading
Always read the diff. Claude sometimes removes things you need, changes function signatures, or adds unwanted dependencies.
💥 No version control
If you're not committing between Claude edits, you have no undo. One bad generation can wreck an hour of work.
💥 Debugging by re-prompting
If Claude's fix doesn't work, don't just say "that didn't work." Paste the actual error. Full stack trace. Every time.
💥 Skipping tests
AI-generated code needs more testing, not less. If Claude wrote it, verify it. Especially edge cases and error paths.
💥 Using Opus for everything
Sonnet handles 80% of dev tasks at comparable quality. Save Opus for architecture, hard bugs, and security reviews.
Building AI Into Your App
🔗 Prompt Chaining
For pipelines: break work into stages. Each Claude call does one focused thing. Pass the output of step 1 as input to step 2. Add validation gates between.
📈 Model Cascading
Use Haiku for fast triage/routing, Sonnet for the main work, Opus only when you need it. This pattern cuts API costs 5-10x at scale.
📚 Structured Output
Always request JSON or a specific format when you need to parse the response programmatically. Define the exact schema in your prompt.
🛡 System Prompts
Give each Claude call a focused role via the system prompt. A "researcher" prompt and a "code reviewer" prompt produce much better results than one generic prompt.
💡 Enable prompt caching if you're sending the same system prompt or context repeatedly. It reduces latency and cost significantly on the Claude API.
